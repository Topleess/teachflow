# TeachFlow · раскадровка объясняющей демонстрации

**Статус:** draft. Это сценарий визуального объяснения, а не доказательство готового продукта.

## Принцип

Scroll-story должна показывать не «набор фич», а одну непрерывную Lesson Session. Анимация работает только после того, как у каждого кадра определены сущность, действие, сохранённый результат и readiness label.

## Черновая последовательность

### Кадр 1. Приглашение

- На экране преподавателя создано занятие.
- Ученик получает ссылку «Присоединиться к уроку».
- Переход по ссылке открывает session без сложного onboarding.
- В первой версии уведомление может быть ручной ссылкой; автоматизация отмечается как hypothesis.

**Смысл:** ученик входит не просто в звонок, а в конкретное занятие.

### Кадр 2. Присутствие

- В workspace появляется видеосвязь.
- На общем учебном поле видны два участника.
- Появляются два цветных курсора с ролями «Преподаватель» и «Ученик».

**Смысл:** участники находятся в одном учебном контексте.

### Кадр 3. Совместное действие

- На canvas находится конкретное задание.
- Для временного технического tracer bullet можно использовать уравнение.
- Преподаватель выделяет часть выражения.
- Ученик переносит элемент или вводит промежуточный ответ.
- Изменение синхронно видно во втором браузере.

**Смысл:** не просто screen share — оба участника действуют.

### Кадр 4. Обратная связь

- Преподаватель отмечает шаг или оставляет короткую пометку.
- Состояние задания меняется без перехода в другой сервис.

**Смысл:** действие и обратная связь принадлежат этому занятию.

### Кадр 5. Сохранение

- Занятие завершается.
- Появляется компактный outcome: выполненное действие, заметка и следующий шаг.
- После reload состояние не исчезает.

**Смысл:** следующий урок начинается не с поиска по чатам и файлам.

### Кадр 6. Продолжение

- Если гипотеза домашней работы подтверждается, ученик получает один следующий шаг.
- Если не подтверждается, кадр заканчивается сохранённым outcome без обещания homework workflow.

**Смысл:** демонстрация не должна продавать неподтверждённую функцию.

## Техническая реализация анимации

Порядок выбора:

1. Сначала кодовый интерактивный прототип с реальным состоянием и двумя браузерами.
2. Затем scroll-driven анимация на базе DOM/SVG/Canvas, использующая те же визуальные состояния.
3. Видео или генеративная анимация допустимы только как mood/storyboard, потому что не доказывают работу продукта.

Возможные средства после выбора архитектуры:

- CSS scroll-driven animations / IntersectionObserver для переходов;
- GSAP ScrollTrigger, если сложность оправдана;
- Lottie/Rive для декоративных микроанимаций;
- кодовый canvas выбранного whiteboard SDK для реальных курсоров и объектов;
- короткая записанная демонстрация реального прототипа как strongest proof.

## Запрещено

- случайные отзывы;
- выдуманные имена учеников как social proof;
- анимация функций без readiness label;
- видео, которое создаёт впечатление рабочего продукта, если показан только концепт;
- одновременно показывать математику, язык, музыку и школу на одном скролле.

## Следующее решение

До финальной раскадровки выбрать:

- реальный предмет/урок для пользовательского кейса;
- технический tracer-bullet activity;
- какое состояние должно сохраняться;
- что в кадрах является working, prototype или concept.
